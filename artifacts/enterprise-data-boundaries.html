Enterprise Data Boundaries As of 2026-06

Per-feature trust-zone diagrams for a Claude Enterprise deployment — where data crosses, where it's processed, where it's stored, and what the BAA covers. For CISOs, security architects, and compliance leads. Every boundary claim traces to governance-overlay.md (the source of truth); verify at privacy.claude.com.
The prompt is the trust boundary. Anything you put in a prompt (or a connected tool result, or an uploaded file) is on Anthropic's infrastructure for the duration of the request — and longer if you use Files API, Memory, or caching. Everything left in your VPC, databases, and source systems stays yours. These diagrams show, per feature, exactly what crosses that line. Architect tool layers to retrieve only what the model must see. [governance-overlay.md §1]

Cross-cutting controls — read these before the per-feature cells

  • No-train ≠ no-store. Commercial default is no-training-on-your-data; it does not mean data isn't stored. Default at-rest retention is 30 days (§11). [§2]
  • ZDR is not default, and not on the Enterprise product UI. Zero Data Retention covers the Messages API + Claude Code only (enterprise agreement, Anthropic approval) — never chat / projects / artifacts. Trust & Safety classifier scores are retained up to 7 years even under ZDR. [§3]
  • inference_geo is first-party-API-only (Opus 4.6 / Sonnet 4.6+). It does not apply to the product UI, to Managed Agents, or to Bedrock/Vertex (those use the hyperscaler region). Workspace geo governs at-rest storage + processing (code exec, image transcoding). [§5]
  • The BAA is not automatic and is per-surface. An Enterprise admin must activate HIPAA compliance + sign. Covered features differ between the first-party API and the Enterprise product surface — the same capability (code execution) is covered on one and not the other. The BAA does not extend to your MCP servers. [§4]
