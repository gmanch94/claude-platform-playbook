◆ Meridian Health illustrative scenario

Throughout this guide we follow Meridian Health — a fictional ~4,000-employee regulated health system standing up Claude on the API surface. Three teams ship: Clinical-AI (patient-record summarization, touches PHI), Revenue-Cycle (claims + coding automation, touches PHI), Data-Science (population analytics, de-identified). A CISO owns governance; a platform lead owns the org; the CFO wants per-team chargeback. Meridian is a worked example, not an Anthropic customer.

WhatA workspace is a container — and only for the API surface

A Console Workspace is the native isolation unit on Claude's build / API surface. It carries four things of its own, and that is the whole mental model:

  • API keys — workspace-scoped; a key belongs to exactly one workspace.
  • Members + roles — who can do what inside this workspace (five roles, below).
  • Limits — its own spend cap and per-model-tier rate limits.
  • Workspace geo — data residency (at-rest storage + endpoint processing), chosen when the workspace is created and immutable afterwards; currently us only. H

What a workspace is not: it does not wrap claude.ai Projects (a seat-product knowledge container) or Cowork — separate surfaces with their own controls (surface-rollout-matrix.html). One nuance: when Claude Code is billed through the Console, Anthropic auto-creates a per-org Claude Code workspace and mints a separate API key for each developer inside it, with its own cap (counting toward the 100-workspace cap like any listed workspace); archiving it disables Console-billed Claude Code sign-in org-wide. So that path is workspace-mediated — the subscription-seat path is not, and hyperscaler traffic (Bedrock / Vertex / Foundry) authenticates and bills through that cloud account, sitting outside the Console workspace model entirely (no workspace, no workspace_id; attribution lives in your cloud project / IAM). A workspace segments programmatic API consumption — by team, environment, or project — nothing else. If a stakeholder says "put the marketing team in their own workspace" but marketing only uses chat and Projects, the workspace won't do what they think.

The hierarchy is organization → workspace → API key. One org (one contract, one bill) contains many workspaces; each workspace contains many keys, members, and limits — drawn as a structure below (org principals and inheritance on top, the workspaces underneath):

Read it top-down: org-level roles cascade into every workspace; workspace-level roles are assigned per workspace. Keys, limits, and geo live in the workspace — never at the org. The Default always exists and can't be removed.

How people get these roles: enterprise SSO (SAML — Team / Enterprise / Console) authenticates the human at login; SCIM / JIT + IdP group-mapping then provisions org membership and role / seat-tier, which cascades into workspaces by the inheritance above. SSO governs the login, not the API credential — keys still authenticate calls, and an already-issued key keeps working until it's revoked, independent of an SSO deprovision. WhyWhy segment at all — five reasons, one principle

Everything you could run under a single key and one workspace. Enterprises don't, for five reasons:

  • Cost attribution — the Usage & Cost API rolls up spend by workspace_id. Make the workspace boundary the budget boundary and per-team cost falls out for free. H
  • Blast-radius containment — a leaked key, a runaway loop, or a bad prompt is scoped to one workspace's keys and one workspace's cap.
  • Access control — five roles, assigned per workspace, keep one team out of another team's keys and traces.
  • Environment isolation — dev / staging / prod as separate workspaces means a prod key is never shared into dev (enterprise-deployment-guide.html §3).
  • Governance posture — residency (Workspace geo) and the real-data-only-in-prod discipline are set per workspace.

The single principle underneath all five: the workspace is the natural budget and blast-radius boundary — draw it once, and attribution, containment, and access follow it.

The Default Workspace is immovable — every org has one, it cannot be renamed, archived, or deleted, and its usage reports back with a null workspace_id. Traffic landing in Default is therefore un-attributable: treat "spend in Default" as an attribution leak to close, not a resting state. H

How · connect & useHow a team gets onto a workspace and starts calling

Once the workspace and its roles exist, a developer connects in five moves. Nothing in application code is workspace-aware — the key carries the workspace.

You — added as a member→ get a workspace-scoped key→ set ANTHROPIC_API_KEY→ call the API→ usage attributes to workspace_id
An admin adds you to the workspace. You can't self-join — an org admin or Workspace Admin adds you as a member with a role (org users and developers are added per workspace). Ask to start as Limited Developer unless you actually need session traces or file download. H
Create a workspace-scoped API key. In that workspace, a Developer or Limited Developer creates a key — Console → the workspace → API keys, or the Admin API. The key belongs to this one workspace; it is what scopes every later call. H
Store the key as a secret. Put it in your app or CI secret store as ANTHROPIC_API_KEY (or send it as the x-api-key header). Never commit it, and never reuse a prod key in dev — one key-set per workspace.
Call the API as normal. Your SDK / HTTP requests with that key run against this workspace and draw on its limits — no other code change. Requests attribute to the workspace's workspace_id in usage reporting. H
Live inside the limits, then confirm attribution. The workspace's rate + spend caps apply under the org ceiling (expect 429 on rate, a hard stop at the spend cap). Need more? Ask the admin to raise the workspace limit (still ≤ org) — not to hand you another key. Then check the Usage & Cost API / dashboard filtered by your workspace_id shows your team's spend, and that nothing landed in Default. H

How · identity & accessThe identity axis — representing people and groups

The credential axis above is machine identity. This is the human one — the SSO / SCIM axis that table pointed at. The principle that governs it: a workspace isolates resources, never people. A group is a real object on Enterprise plans — a named collection of members you can create natively (Org settings → Groups) or sync from your identity provider (Okta, Entra) over SCIM. It is its own object, not a field on a user; membership is many-to-many, and a member's role + spend cap are derived through it. Represent a group by deciding what it is for: a permission tier (→ a role) or a budget / blast-radius boundary (→ a workspace). Keep those two axes separate.

Two role systems reach Claude — don't conflate them. The Console (API / platform) side and the Enterprise seat (claude.ai) side carry different built-in role sets:

Built-in seat roles vs Custom

The four built-in Enterprise seat roles — Primary Owner, Owner, Admin, User — carry fixed access and are unaffected by the capability system. To put a member under fine-grained RBAC you switch their role to Custom; only then do groups + capabilities (features, connectors, models, admin-permission areas) govern what they can do. Built-in and Custom coexist, so migration is gradual. One consequence worth knowing: you can't layer extra admin permissions onto a built-in role — to delegate (say) billing without handing over Owner, move the person to a Custom role in a group that grants exactly that. H

Groups + capabilities are additive

Enterprise groups are real objects — they even carry their own group spend limits — and they're how members reach custom roles: member → group → role. Capabilities compose additively, which drives the recommended shape: one base "Standard Access" role for everyone (web search, memory, projects) assigned through an "All Users" group, then thin additive roles (a "Cowork Enabled" role that only adds Cowork) layered via extra groups. No conflict resolution, no per-person toggling. H

Make the IdP the source of truth

You can create groups natively in Console, but the durable pattern is to source them from your IdP — author membership once, and let SCIM provision and deprovision it:

  • One IdP group per role, and (unless you're single-seat) one per seat tier.
  • Map those IdP groups to "Custom" roles in the role-mappings table (Org settings → Organization and access). Provisioning then sets each member's role from their group membership. H
  • SCIM directory-sync handles add and remove automatically — deprovisioning on offboarding is the security win; JIT instead provisions on first SSO login.

How · day-2 operationsRunning the fleet without attribution debt

  • Watch the Default Workspace. Its usage is null workspace_id — un-attributable. A key created there by accident is silent spend. Sweep for it.
  • Showback before chargeback. Meter (workspace = team) → show teams their monthly spend for two clean cycles → only then move money to team budgets (token-budget-governance.html).
  • Cap experiments, alert prod. Hard caps on dev/sandbox (harmless brown-out); alert thresholds + escalation on prod (never a cliff). A blanket org cap browns out prod in the month's last week.
  • Least privilege by default. New members → Limited Developer; elevate to Developer/Admin only when the trace/exfil access is actually needed.
  • One key-set per workspace, rotated independently. Never share a prod key into dev — a dev leak then equals prod exposure plus blended attribution.

How · decommissionArchiving is a ritual, not a button

Because archive cascades to every key and cannot be undone, treat it as a decommission ritual:

  • 1 · Drain — move or retire every workload using the workspace's keys; confirm zero live traffic.
  • 2 · Export — pull any usage/cost history you need for the record (the Console keeps data, but capture what your FinOps or audit trail requires).
  • 3 · Archive — then archive, in Console or via POST …/archive. Keep an out-of-band changelog of who/when/why (the Console retains data, not your rationale).

"Delete my data" is a retention question, not a workspace op. Archiving removes the operational container; it does not purge the data. A stakeholder who needs the content gone is asking for a retention / ZDR / Compliance-API-delete control (governance-overlay.html), not an archive. Conflating the two is how a DSAR or eDiscovery-deletion obligation quietly fails.
