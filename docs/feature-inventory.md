**Enterprise identity & access row (added 2026-07-15, [H] live-verified this session)** — the *human-auth* counterpart to WIF's machine-auth. **Two distinct role systems, don't conflate:** Console/API **org** roles (User, Claude Code User, Limited Developer, Developer, Billing, Admin — support.anthropic.com/en/articles/10186004-api-console-roles-and-permissions) vs **Enterprise claude.ai seat** roles — built-in **Primary Owner / Owner / Admin / User**, plus **Custom**, which puts a member under the group→capability RBAC (features / connectors / models / admin-permission areas). Built-in roles keep fixed access and can't be layered with extra admin perms; only Custom members are governed by groups. **Groups** are a first-class Enterprise object — created natively (Org settings → Groups) *or* IdP-synced over SCIM (best practice: IdP-sourced, for auto-deprovisioning), never a user-level attribute — they carry their own **group spend limits**, and reach roles via *member → group → role*; capabilities are **additive** (a base "Standard Access" role + thin additive roles like "Cowork Enabled"). **SCIM/JIT = Enterprise + Console-org only** (Team gets SSO but not SCIM; consumer plans neither); map IdP groups → "Custom" roles in the role-mappings table (Org settings → Organization and access). Top failure: **SCIM/SAML email-attribute mismatch** (Okta `user.login` vs `user.email`; Entra syncs only ~every 40 min). **Dual-seat plans:** a custom role can't override a seat-level restriction — the seat is the ceiling. Sources verified live 2026-07-15: identity-management collection (support.claude.com/en/collections/17270717), role-based-permissions (article 13930458), JIT/SCIM provisioning (article 13133195). Rendered into the identity axis (`#identity`) of [`../artifacts/enterprise-workspaces-guide.html`](../artifacts/enterprise-workspaces-guide.html).

**Messages API key row (added 2026-07-19)** — the primary *inference* credential (workspace-scoped `sk-ant-api…`), distinct from the **Admin** key (org mgmt, #78) and **WIF** short-lived `sk-ant-oat01-` tokens (#85). **Usage-tier (rate-limit) row (added 2026-07-15, [H] live-verified)** — the fifth "tier" and the one with no prior inventory row. Org-level bracket **Start → Build → Scale** (+ **Custom** when account-team-managed), measured in RPM / ITPM / OTPM. **Anthropic-set** (service-configured) — auto-advances with spend; you can *request* an increase in Console at ≥50% utilization but can't self-assign a tier. **Claude Platform on AWS is pinned to Start and does not auto-advance** (increases via account rep only; the self-serve request flow is disabled). Distinct from **service tiers** (priority/standard/batch) and from **spend caps** (which attach to org / workspace / group / member, never a role). The full five-way "tier" disambiguation matrix lives in [`../artifacts/claude-misconceptions.md`](../artifacts/claude-misconceptions.md) §6. Source: docs.claude.com/en/api/rate-limits.

**Token economics** — see [`../artifacts/cost-calculator.html`](../artifacts/cost-calculator.html). Cache read ≈ 10% of input. Cache write 5m ≈ 125% of input. Cache write 1h ≈ 200% of input. Batch ≈ 50% of all rates.

**Minimum cacheable prompt** — Opus 4.8 lowered to **1,024 tokens** (was 4,096 on Opus 4.7 / 4.6). Sonnet 4.6 + Haiku 4.5 are 1,024. **Sonnet 5's floor isn't stated in launch materials — verify rather than assuming 1,024 carries over**, especially given its tokenizer change (same input maps to more tokens). Prompts shorter than the floor are processed uncached with no error. Lower floor on 4.8 = more small prompts become cache-eligible. See [`../artifacts/data-advisory.md`](../artifacts/data-advisory.md) cache-eligibility shape.

**Extended thinking note** — Opus 4.7 and 4.8 support **adaptive thinking only**: `thinking: {"type": "adaptive"}` plus the `effort` parameter to control depth. Setting `thinking: {"type": "enabled", "budget_tokens": N}` returns HTTP 400. They also reject non-default `temperature` / `top_p` / `top_k` (400). Opus 4.6 and earlier: manual `budget_tokens` still works but adaptive is recommended.

**Fast mode (Opus 4.8)** — research preview on the Claude API. Lower-latency response path. Preview status — do not pin production SLAs to it. Doc anchor: docs.claude.com/en/docs/about-claude/models/whats-new-claude-4-8.
